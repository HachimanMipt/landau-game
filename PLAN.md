# PLAN: MVP веб-игры «Теорминимум Ландау»

## 1. Краткое резюме архитектуры

Выбран минимальный монолит без лишних зависимостей, но с более простой внутренней моделью UI:

- `FastAPI` как HTTP-приложение и точка маршрутизации.
- `Jinja2` для серверного HTML.
- `SQLite` + `SQLAlchemy 2` для хранения участников, прохождений и ответов.
- `Pydantic` для валидации редактируемого контента карточек в `data/questions.json`.
- Небольшой card engine: сервер собирает текущую карточку из данных и состояния прохождения.
- Два переиспользуемых типа карточек:
  - `decision_card` для вопроса и ответа студента;
  - `landau_card` для реплики персонажа, разбора и исторической карточки.
- Чистый `JavaScript` только для свайпа, анимаций и отправки решения без фронтенд-сборщика.
- `pytest` + `TestClient/httpx` для unit/integration-тестов.

Ключевой принцип: сервер является единственным источником истины для состояния игры. Клиент не хранит критические данные о прогрессе, правильности или следующей карточке. В cookie хранится только идентификатор текущего `GameRun`, а backend сам решает, какую карточку сейчас рендерить.

## 2. Принятые решения по неоднозначностям

1. Стартовый экран и регистрация в ТЗ частично дублируют друг друга.
   Решение: сделать два шага.
   `GET /` показывает вступление и CTA.
   `GET /register` показывает форму `имя + команда`.

2. В ТЗ указано 10 вопросов, но допускается будущий пул и случайный выбор.
   Решение: MVP использует 10 активных вопросов в фиксированном порядке, но `GameRun` хранит `question_order_json`, чтобы позже можно было перейти к пулу без пересборки модели.

3. Страница `review` упомянута в тестах и примерной структуре, но не раскрыта в пользовательском сценарии.
   Решение: `review` будет доступна только после завершения игры и покажет полный разбор всех 10 работ.

4. Исторические факты могут быть демонстрационными и не всегда отредактированными.
   Решение: в контент добавить `sources` и `fact_check_status`, чтобы отличать подтвержденные факты от временных заготовок.

5. В ТЗ нет отдельного описания admin-аккаунтов.
   Решение: для MVP использовать один пароль администратора из `.env`, хранить факт входа в подписанной cookie-сессии.

6. В ТЗ нет явного ответа, должен ли проект работать без JavaScript.
   Решение: свайп и анимации требуют JS, но резервные кнопки должны работать через обычный HTML POST как функциональный fallback.

7. Вместо отдельных сложных экранов игры нужен простой редактируемый набор карточек.
   Решение: публичная игровая часть строится вокруг одного маршрута `/play`, одного общего контейнера и двух шаблонов карточек.

## 3. Основные допущения

- Приложение запускается для одного мероприятия на одном ноутбуке в локальной сети.
- Достаточно одного процесса `uvicorn`; многопроцессный запуск для SQLite в MVP не нужен.
- 30 одновременных участников достижимы для SQLite при коротких транзакциях, `WAL` и отсутствии тяжелых фоновых задач.
- Игра полностью русскоязычная.
- Имена участников не уникальны; идентификация идет по `UUID` прохождения.
- Демонстрационный набор вопросов хранится в репозитории и загружается при старте, без CMS.
- В MVP не добавляются WebSocket, Redis, Celery, Docker, frontend-build tools и SPA-фреймворки.

## 5. Модели данных

### 5.1. SQLAlchemy модели

#### `Participant`

- `id: int` PK
- `public_id: UUID` unique, index
- `name: str(40)` not null
- `team: str(40) | null`
- `created_at: datetime`

Назначение: карточка участника события. Имя не используется как идентификатор.

#### `GameRun`

- `id: int` PK
- `public_id: UUID` unique, index
- `participant_id: FK -> Participant.id`
- `status: enum("in_progress", "completed")`
- `screen_state: enum("question", "result", "final")`
- `started_at: datetime`
- `completed_at: datetime | null`
- `current_position: int`
- `score: int`
- `question_order_json: str`
- `current_question_shown_at: datetime | null`
- `last_answered_question_id: str | null`
- `updated_at: datetime`

Обоснование дополнительных полей:

- `screen_state` нужен, чтобы после перезагрузки reliably показывать не только текущий вопрос, но и несмотренный результат.
- `question_order_json` сохраняет фактический порядок вопросов для конкретного прохождения.
- `current_question_shown_at` позволяет считать время ответа на сервере, а не доверять клиенту.
- `last_answered_question_id` упрощает получение последнего результата при `screen_state = "result"`.

#### `Response`

- `id: int` PK
- `game_run_id: FK -> GameRun.id`
- `question_id: str`
- `selected_verdict: enum("correct", "inaccuracy")`
- `is_correct: bool`
- `swipe_direction: enum("left", "right")`
- `input_method: enum("swipe", "button", "keyboard")`
- `shown_at: datetime`
- `answered_at: datetime`
- `response_time_ms: int`

Ограничения:

- unique(`game_run_id`, `question_id`)
- `response_time_ms >= 0`

Примечание: фактическая память о жесте хранится на сервере в `swipe_direction`, а игровая семантика хранится в `selected_verdict`. При фиксированном mapping это одно и то же решение, но хранить оба поля полезно для аналитики и отладки жестов.

### 5.2. Контент сцен и карточек

В `data/questions.json` хранится не просто вопрос, а редактируемая сцена игры. Одна сцена разворачивается в две карточки:

- `decision_card` с текстом задания и ответом студента;
- `landau_card` с разбором, репликой Ландау и историческим блоком.

В БД вопрос как отдельная таблица не нужен, потому что для MVP контент read-only и контролируется репозиторием.

### 5.3. Внутренняя view-модель карточки

Сервер на каждом шаге собирает `CardViewModel`, который рендерится единым маршрутом `/play`.

Минимальная структура:

- `card_type: "decision" | "landau"`
- `card_id: str`
- `title: str | null`
- `body: str | null`
- `meta: dict`
- `swipe_enabled: bool`
- `left_label: str | null`
- `right_label: str | null`
- `portrait_image_url: str | null`
- `primary_action_label: str | null`

Это позволяет менять текст и наполнение карточек без переписывания шаблонов.

## 6. Схема пользовательского состояния

Состояние прохождения определяется только сервером:

1. Нет активной cookie или `GameRun` не найден.
   Действие: редирект на `/`.

2. `status = in_progress`, `screen_state = question`.
   Действие: показать текущую `decision_card`.

3. `status = in_progress`, `screen_state = result`.
   Действие: показать текущую `landau_card`, собранную по `last_answered_question_id`.

4. `status = completed`, `screen_state = final`.
   Действие: показать финальный экран.

5. `status = completed` и пользователь открывает `/review`.
   Действие: показать полный разбор прохождения.

### Переходы

- Регистрация -> создается `Participant` + `GameRun`, cookie привязывается к `GameRun.public_id`, состояние `question`, `current_position = 1`.
- Открытие вопроса -> если `current_question_shown_at` пустой, сервер заполняет его текущим временем.
- Ответ на вопрос -> создается `Response`, пересчитывается `score`, `screen_state` меняется на `result`, `last_answered_question_id` обновляется.
- Кнопка "Следующая работа" -> если это не 10-й вопрос, `current_position += 1`, `screen_state = question`, `current_question_shown_at = now`.
- После 10-го результата -> `status = completed`, `screen_state = final`, `completed_at = now`.

### Защита от смены URL

- В публичном URL нет номера вопроса.
- Маршрут `/play` сам решает, что показывать.
- Даже если пользователь откроет `/review` до завершения, сервер отдаст `403` или редирект на `/play`.

## 7. Маршруты приложения

### 7.1. Публичные HTML-маршруты

- `GET /`
  Вступительный экран, концепция, реплика Ландау, кнопка перехода к регистрации.

- `GET /register`
  Форма регистрации.

- `POST /register`
  Валидация формы, создание `Participant` и `GameRun`, установка cookie, редирект на `/play`.

- `GET /play`
  Единая точка входа в текущий игровой экран.
  В зависимости от состояния рендерит `play.html` с `decision_card` или `landau_card`, либо `final.html`.

- `POST /play/answer`
  Fallback-маршрут для резервных кнопок без JS.
  Сохраняет ответ и делает редирект обратно на `/play`.

- `POST /play/next`
  Переход от карточки результата к следующему вопросу или финалу.

- `GET /review`
  Полный разбор завершенного прохождения.

### 7.2. API-маршруты

- `POST /api/answers/current`
  JSON endpoint для свайпа и клавиатурного ответа.
  Принимает:
  - `selected_verdict`
  - `input_method`

  Возвращает:
  - `ok`
  - `redirect_url`
  - минимальный payload для клиентской блокировки и обработки ошибки

  В API не передаются `is_correct`, `score`, `correct_verdict`, `shown_at`, `answered_at`.

### 7.3. Административные маршруты

- `GET /admin/login`
- `POST /admin/login`
- `POST /admin/logout`
- `GET /admin`
  Сводная статистика по участникам, среднему баллу, времени, частым ошибкам.

- `GET /admin/export.csv`
  Выгрузка по участникам и ответам.

- `POST /admin/reset`
  Сброс результатов только после явного подтверждения в форме.

Опционально, если для графиков окажется удобнее отдельный источник данных:

- `GET /admin/stats.json`

Но на первом проходе предпочтительнее обойтись серверным рендерингом без лишнего API.

## 8. Формат `data/questions.json`

### 8.1. Итоговая структура записи

```json

  "position": 1,
  "decision_card": {
    "work_title": "Работа 1 из 10",
    "question_text": "Почему летом в Северном полушарии теплее, чем зимой?",
    "student_answer": "Летом Земля находится ближе к Солнцу и поэтому получает больше тепла.",
    "swipe_left_label": "Есть неточность",
    "swipe_right_label": "Ответ верный"
  },
  "correct_verdict": "inaccuracy",
  "landau_card": {
    "verdict_title_correct": "Вы заметили неточность",
    "verdict_title_wrong": "Здесь неточность осталась незамеченной",
    "explanation": "Главная причина смены сезонов — наклон земной оси...",
    "inaccuracy_explanation": "Расстояние до Солнца не является главной причиной...",
    "character_quote": "Уверенный тон не исправляет неверную причину.",
    "fact_title": "Физическое мышление",
    "fact_text": "Этот блок хранит историко-научную реплику, связанной с темой вопроса.",
    "place": null,
    "period": null,
    "people": null,
    "importance": "Пример показывает различие между корреляцией и причиной."
  },
  "sources": [],
  "difficulty": 1,
  "fact_check_status": "needs_review",
  "enabled": true
}
```

### 8.2. Правила валидации

- В MVP должно быть ровно 10 записей с `enabled = true`.
- `id` уникален.
- `position` уникален среди активных вопросов.
- `decision_card` и `landau_card` обязательны.
- `correct_verdict` принадлежит множеству `{"correct", "inaccuracy"}`.
- Обязательные поля внутри обеих карточек не пусты после `trim`.
- `difficulty` ограничен диапазоном `1..3`.
- `sources` всегда массив строк.
- `fact_check_status` принадлежит множеству `{"verified", "needs_review"}`.

### 8.3. Контентные замечания

- `decision_card` хранит только редактируемый текст вопросной карточки.
- `landau_card` хранит только редактируемый текст карточки персонажа и разбора.
- На клиент отдается только текущая собранная карточка.
- `correct_verdict`, `landau_card.explanation` и `landau_card.inaccuracy_explanation` не должны попадать в HTML до момента ответа.

## 9. Визуальная и UI-архитектура

- Общий макет строится как единый "стол / архивная папка", а не набор несвязанных страниц.
- Для публичной части используется один визуальный каркас (`base.html`) и один игровой контейнер (`play.html`) с двумя внутренними шаблонами карточек.
- Цвета и типографика оформляются через CSS custom properties.
- Иллюстрация Ландау вставляется только через один путь:
  `app/static/images/landau-placeholder.svg`
  Это позволяет потом просто заменить файл без правки шаблонов.
- Смысл направления должен читаться не только по цвету, но и по подписи, иконке и форме.
- `decision_card` и `landau_card` должны иметь одинаковую внешнюю "рамку", чтобы интерфейс ощущался как одна стопка карточек, а не как разные страницы.

## 10. Механика свайпа и резервного ввода

### 10.1. Правило определения жеста

- До явного преобладания горизонтального движения нативный вертикальный скролл не блокируется.
- Горизонтальный drag начинается только если:
  - `abs(dx) >= horizontal_start_threshold`
  - `abs(dx) > abs(dy) + vertical_guard_threshold`
- После перехода в режим drag вызывается `preventDefault`, карточка двигается и поворачивается.

### 10.2. Защита от двойной отправки

Клиент:

- сразу ставит лок `isSubmitting`;
- отключает pointer events;
- отключает клавиатурные обработчики;
- дизейблит резервные кнопки;
- игнорирует повторные gesture/button events до ответа сервера.

Сервер:

- принимает ответ только если `screen_state = question`;
- сверяет, что вопрос соответствует `current_position`;
- вставляет `Response` под уникальным ограничением;
- при повторной отправке возвращает безопасный ответ без повторной записи.

## 11. Подробный план реализации Phase 1-6

## Phase 1. Каркас проекта и инфраструктура

Задачи:

- создать `pyproject.toml`, `.env.example`, `.gitignore`;
- поднять FastAPI-приложение, конфиг, Jinja, статику;
- настроить SQLAlchemy engine/session factory;
- включить SQLite `WAL`, `foreign_keys = ON`, разумный `busy_timeout`;
- подготовить `scripts/init_db.py`;
- добавить базовые шаблоны и пустые маршруты;
- зафиксировать README с установкой и запуском.

Критерии готовности:

- приложение стартует локально;
- база и таблицы создаются одной командой;
- `/`, `/register`, `/play`, `/admin/login` отдают ожидаемые ответы/редиректы;
- README содержит минимальный рабочий путь запуска.

## Phase 2. Контент и доменная логика

Задачи:

- описать `Pydantic`-схему сцены с двумя вложенными карточками;
- реализовать загрузчик и валидацию `questions.json`;
- подготовить 10 демонстрационных вопросов;
- реализовать `CardViewModel` builder для `decision_card` и `landau_card`;
- реализовать enums и чистые функции:
  - определение правильности ответа;
  - подсчет итогового балла;
  - расчет финального статуса.

Критерии готовности:

- невалидный `questions.json` валит старт приложения с понятной ошибкой;
- ровно 10 активных вопросов успешно загружаются;
- обе карточки каждой сцены собираются без ручной логики в шаблонах;
- unit-тесты контента и доменной логики проходят.

## Phase 3. Backend игрового цикла

Задачи:

- реализовать регистрацию участника;
- создать и сохранять `GameRun`;
- реализовать серверное восстановление после refresh;
- реализовать сохранение `swipe_direction`, ответа, времени показа и времени ответа;
- запретить повторный ответ, пропуск вопросов и продолжение завершенного run;
- реализовать `/play`, `/play/answer`, `/play/next`, `/api/answers/current`;
- собирать текущую карточку на сервере через `cards.py`, а не через ветвистые шаблоны.

Критерии готовности:

- после refresh пользователь возвращается ровно в нужное состояние;
- нельзя ответить дважды на один и тот же вопрос;
- нельзя ответить вне текущей позиции;
- после десятого вопроса открывается финал;
- route/integration-тесты на основной цикл проходят.

## Phase 4. Игровой интерфейс и мобильное поведение

Задачи:

- сверстать экраны `start`, `register`, `play`, `final`, `review`;
- сверстать два переиспользуемых шаблона: `decision_card` и `landau_card`;
- реализовать свайп на Pointer Events;
- добавить резервные кнопки и поддержку клавиатуры;
- добавить анимации drag / snap-back / fly-out / result reveal;
- обеспечить адаптивность от `320px`;
- решить конфликт длинного текста, внутреннего скролла и горизонтального жеста;
- оформить доступность: focus state, ARIA labels, live region для результата.

Критерии готовности:

## 12. Стратегия тестирования

### 12.1. Автоматические тесты

`test_content.py`

- загружаются ровно 10 активных вопросов;
- `id` уникальны;
- `position` уникальны;
- обе вложенные карточки валидны;
- `correct_verdict` валидны;
- обязательные поля заполнены.

`test_game_logic.py`

- правильность считается сервером;
- счет обновляется корректно;
- финальная категория определяется правильно;
- завершенный run нельзя продолжать;
- направление свайпа сохраняется корректно;
- повторное прохождение создает новый run и не меняет старый.

`test_public_routes.py`

- регистрация работает;
- пустое имя отклоняется;
- игра без cookie редиректит на старт;
- после ответа `play` переключается с `decision_card` на `landau_card`;
- после десятого ответа доступен финал;
- `review` недоступен до завершения.

`test_answer_api.py`

- JSON endpoint принимает только допустимые verdict/input_method;

- нельзя отвечать вне текущего вопроса;
- `swipe_direction` записывается корректно;
- `input_method` записывается корректно.

`test_admin.py`

- dashboard закрыт без логина;
- верный пароль дает доступ;
- неверный пароль не дает доступ;
- CSV формируется;
- reset требует подтверждения.

### 12.2. Ручной checklist

- iPhone Safari: свайп в обе стороны, кнопки, длинный текст, поворот экрана не обязателен.
- Android Chrome: тот же сценарий.
- Desktop Chrome/Safari/Firefox: мышь, кнопки, клавиатура.
- Перезагрузка на вопросе.
- Перезагрузка на результате.
- Попытка двойного тапа/свайпа.
- Проверка на ширине `320px`.

### 12.3. Что не добавляем в MVP

- Playwright и полноценные e2e browser tests.
- Frontend test runner только ради одного swipe-файла.

Если логика определения свайпа получится изолированной и короткой, ее можно вынести в чистые функции и покрыть lightweight-проверками позже, но это не блокер MVP.

## 13. Технические риски и способы их снижения

### Риск 1. SQLite блокировки при 30 участниках

Снижение:

- `WAL` режим;
- короткие транзакции;
- один процесс приложения;
- `busy_timeout`;
- минимизация тяжелых admin-операций во время игры;
- smoke-test на множественные регистрации и ответы.

### Риск 2. Двойная отправка ответа

Снижение:

- клиентский лок;
- серверная проверка `screen_state`;
- unique constraint на `(game_run_id, question_id)`;
- PRG/fetch->redirect flow без повторной мутации при refresh.

### Риск 3. Неверное восстановление после перезагрузки

Снижение:

- поле `screen_state`;
- поле `current_question_shown_at`;
- авторитетный маршрут `/play`;
- тесты на refresh в состояниях `question`, `result`, `final`.

### Риск 4. Утечка правильного ответа в клиент

Снижение:

- не встраивать `correct_verdict`, `explanation`, `inaccuracy_explanation` в HTML вопроса;
- не отдавать полный `questions.json` на клиент;
- API принимает только выбор игрока, а не "правильность".

### Риск 5. Конфликт горизонтального свайпа и вертикального скролла

Снижение:

- жест начинается только при явном преобладании `dx`;
- длинный текст живет во внутреннем scroll-контейнере карточки;
- до lock-in по горизонтали скролл страницы/карточки не блокируется;
- ручной mobile checklist обязателен.

### Риск 6. Слишком сложный UI для MVP

Снижение:

- UI остается серверным;
- публичный gameplay сводится к двум типам карточек и одному маршруту;
- JS отвечает только за жест, блокировку повторной отправки и простые анимации;
- без SPA, без client-side state management library.

### Риск 7. Смена иллюстрации Ландау позже сломает шаблоны

Снижение:

- один стабильный путь к изображению;
- шаблоны не завязаны на конкретный формат кроме обычного `<img>`.

## 14. Команды проверки результата

Предлагаемый порядок локальной проверки:

1. `python3 -m venv .venv`
2. `. .venv/bin/activate`
3. `pip install -e .[dev]`
4. `cp .env.example .env`
5. `python scripts/init_db.py`
6. `pytest`
7. `uvicorn app.main:app --host 0.0.0.0 --port 8000`
8. Открыть `http://127.0.0.1:8000`
9. С телефона открыть `http://<LOCAL_IP_НОУТБУКА>:8000`

Дополнительно в README нужно пояснить:

- как узнать локальный IP;
- почему guest Wi-Fi или client isolation могут мешать доступу с телефона;
- почему `127.0.0.1` работает только на самом ноутбуке.

## 15. Критерии готовности всего MVP

MVP считаю готовым, когда одновременно выполняются условия:

- проект ставится по одной документированной последовательности команд;
- локальный запуск стабилен;
- регистрация работает;
- 10 вопросов проходят по фиксированному сценарию;
- весь игровой поток собирается из двух шаблонов карточек;
- свайп и резервные кнопки работают на телефоне;
- каждый ответ сохраняется один раз;
- после каждого ответа показывается корректно собранная `landau_card` с разбором;
- после 10-го вопроса показывается итог;
- refresh не ломает прогресс;
- admin видит сводную статистику и может скачать CSV;
- основные тесты проходят;
- интерфейс пригоден при `320px`;
- в клиент не утекает правильный ответ до выбора игрока.

## 16. Самостоятельно принятые решения

- Выбран двухшаговый сценарий `start -> register`, а не единая форма на одном экране.
- Добавлены поля `screen_state`, `question_order_json`, `current_question_shown_at`, `last_answered_question_id` в `GameRun`.
- Страница `review` трактуется как послематчевый полный разбор.
- Для фактов добавлен `fact_check_status`.
- Для надежного fallback предусмотрен отдельный HTML POST помимо JSON API.
- Игровой UI упрощен до двух типов карточек и одного маршрута `/play`.
- Память о жесте хранится сервером как `swipe_direction` вместе с игровым вердиктом.
